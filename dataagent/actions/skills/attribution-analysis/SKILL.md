# 通用归因分析 Skill

本 skill 是**路由器**：先判断口径（A/B/C），再按 Phase 0–7 路由到对应 reference 子文件。详细方法、输出边界与执行级硬规则都在 `references/` 里，按需读取——**不要把本文件当成方法手册通读**。

## 1. 口径分类（先做这一步）

| 分支 | 解释对象 | 口径 | 核心问句 | 输出形态 |
|------|--------|------|---------|---------|
| **A. 指标异动归因** | 真实世界（指标变化或群体差异） | 变化 / 差异分解 | "指标为什么变了 / 为什么 X 群体高于 Y 群体" | 贡献因子 + 候选解释 |
| **B. Contribution Attribution（功劳归因）** | 真实世界（**已发生结果**由多个参与因素共同形成；营销是重要应用） | 功劳 / 信用分配 | "在既定结果下，谁贡献了多少功劳" | 功劳份额、贡献金额/数量、贡献排序、可选决策建议 |
| **C. 因果归因** | 真实世界（干预与结果之间的因果链） | 增量效应 | "干预导致了多少因果增量" | 效应估计 + 置信区间 |

**不同口径的"归因"是不同的问题**，不可混用：

- "功劳分配"（B）≠ "变化解释"（A）≠ "增量效应"（C）
- A 的"贡献度"是数学分解，不是因果结论
- B 的"功劳"默认是相关性或模型依赖视角，除非经 C 校准
- **任何 A/B 结论都不得表述为因果**；C 结论须写明识别假设与适用边界

### 不解决什么

- 纯异常检测 / 告警阈值设计（只处理"检测到异常后如何解释"）
- 严格因果图结构学习（仅使用已确定的因果图）
- 实验设计本身（仅分析实验结果）
- 预测建模 / 特征选择（只做归因解释，不做预测）

## 2. 默认工作流（Phase 0–7）

> **Phase 是思考顺序，不是 todo 清单。** 有结构化契约时，主估计出来后按 Phase 7 输出 JSON；todo 完成、过程摘要不计交付。

> **复杂度闸门**：任务已说清类别、给齐可用变量/实验设计、或只是了解背景不据此改业务 → 用 1–2 句交代口径后**直接进 Phase 3**；否则完整走 Phase 0–2。任务已具体时，长篇"流程作文"不提高可信度，反而冲淡要点。

- **Phase 0 可行动性审查**：结论若**不会**改预算/投放/产品/模型 → 简化输出；若会据此行动 → 必须写清**用什么数据或实验复核**。
- **Phase 1 问题对齐**：明确要解释哪一项结果、怀疑哪些原因、什么时间/粒度、做完支撑什么动作。详见 [00](references/core/00_context_discovery.md)。
- **Phase 2 分类与路由**：判定主分支 A/B/C（可叠加）+ 子类型。详见 [01](references/core/01_problem_classification.md)。
- **Phase 3 数据准备与口径固定**：数据质量预检 → 口径对齐 → 归因事实表。**若数据缺陷大到无法区分噪声与业务 → 停止归因，只交付数据问题与修复建议**。详见 [02](references/core/02_data_preparation.md)。
- **Phase 4 方法选型与执行**：先做**简单、可解释**的基线（规则拆解 / 浅层回归 / 已有模型 + SHAP）暴露口径与数据问题；再按下方分支表路由。结论要用于决策时，说明用什么实验或独立对照复核。
- **Phase 5 推断与不确定性**：给点估计 + 必要的不确定性；敏感性/稳健性用于提升结论强度，不阻塞结构化输出。详见 [50](references/core/50_inference_uncertainty.md)。
- **Phase 6 评估与校准**：当结论要支撑高风险决策或需要提升结论强度时，用独立思路对照（如 MTA 与 MMM、或归因与实验）；方向一致增信，不一致须写明原因。详见 [51](references/core/51_evaluation_calibration.md)。
- **Phase 7 输出护栏**：按调用方契约输出；若契约要求 JSON，只输出 JSON。详见 [52](references/core/52_output_guardrails.md)。

## 4. 输出边界

- **调用方契约优先**：调用方给出 JSON schema、字段、机器可读块或其他结构化契约时，只按该契约输出。
- **主估计锁定即可交付**；稳健性、互核、校准未完成时只降低结论强度，不阻塞输出。
- **过程不算交付**：计划、todo、工具回执或过程摘要不算契约交付物。
- 本 skill **不定义** JSON、schema、字段名、字段顺序或字段是否必填。这类定制化输出契约必须放在调用方 prompt、评测脚本或用户指令中。

## 5. 全局护栏

整个分析过程必须遵守：

1. **三口径不可混用且 A/B 非因果**：A（变化解释）/ B（功劳分配）/ C（增量效应）是不同问题；A/B 结论不得表述为因果增量，除非经 C 分支识别流程；C 结论须写明识别假设与适用边界。
2. **不同拆解维度（地区 vs 渠道 vs 人群）是不同切片上的占比**，不可直接加总或比大小论对错；头部项高度重合多半是同一现象的两种看法。
3. **独立对照用于提高结论强度，不是所有任务的交付前置**；若只有一种思路，须写明并降低结论强度。若调用方要求结构化输出，主估计锁定后先按契约交付，未完成的对照不阻塞输出。
4. **输入不足时降低结论强度而非强行量化**；缺数据就说没数据，不编造。
5. **分母过小、口径变化、样本不足会失真**——遇到必须在输出中标注。
6. **B 分支内部路由**：有用户/会话级触点轨迹 → 21（MTA）；仅渠道/区域/周月花费与结果汇总且目标是聚合贡献/预算配比 → 22（MMM）；其他资源/功能/流程贡献 → 20a；KPI 涨跌、按维度下钻、无功劳分配语义 → A。
7. **用归因指导预算时须写明反馈回路**：出价与分配会因结论调整，未来数据分布会变，静态归因不是永续真理。
8. **若没有增量实验或等价对照**，B 结论只能写成相关意义上的功劳拆分或模型依赖贡献，**不得**写成因果增量 ROI / 因果贡献。
9. **维护"假设台账"**：每个主结论至少对应 1 条关键假设，标注状态（已验证 / 部分验证 / 未验证）与潜在影响。
10. **每个主要结论必须可追溯到对应证据**（数据观察 / 统计结果 / 实验 / 外部事实）；证据不足时只能作为候选解释，不得表述为确定结论。
