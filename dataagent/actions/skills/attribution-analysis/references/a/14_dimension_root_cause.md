# A.4 维度根因定位

本文件是 A 分支的横向增强层。它不负责“公式怎么拆”，而负责回答“异常集中在哪些维度值或维度组合”。可叠加在 [11_additive_metrics.md](11_additive_metrics.md)、[12_ratio_derived_metrics.md](12_ratio_derived_metrics.md)、[13_nonlinear_composite_metrics.md](13_nonlinear_composite_metrics.md) 之上。

## 1. 何时使用

- 已经知道指标为什么会变一部分，但还需要回答“到底是哪个维度 / 哪个维度组合最可疑”
- 候选维度不少于 2 个
- 问题已经从“公式拆解”转向“attribute-value combinations 搜索”

## 2. 输入要求

- 来自主路径的目标异常定义
- 可切片的维度表
- 每个维度值或维度组合对应的 actual / expected，或 before / after
- 最低样本量 / 最低支持度阈值

## 3. 问题本质与方法边界

维度根因定位的本质是搜索 attribute-value combinations。难点不在于“有没有一个最强单点”，而在于：

- 搜索空间巨大
- root cause 可能是一个 set，不一定是单个维度值
- 不同维度视角可能只是同一现象的不同投影

默认输出：候选集合；不输出唯一真因。

## 3.1 与「打分」相关的最简可核对式

**（1）可加总 KPI 下的切片闭合（与 11 同族）**

**叶子单元** `j`：当前分析里选定的**最细可加总切片**（多维指标立方体上的最小聚合格；与 HotSpot 等文献中的 **LEAF** 同义）。**不是**决策树模型里的「叶节点」。

设这样一组 `j` 互斥且穷尽，对比两列为 `v_j` 与 `hat_v_j`（例如 actual / expected，或当期 / 基期；**与总体口径一致**）。记单格偏差 `delta_j = v_j - hat_v_j`。再记总体聚合

```
# sum_j(·) 表示对叶子下标 j 求和（Σ_j）

V     = sum_j( v_j )
V_hat = sum_j( hat_v_j )
```

当且仅当 **`V` 等于业务上的总体实际值、`V_hat` 等于同一口径下的总体对比值**（同一套 `j` 划分上可加总还原总体）时，有恒等式

```
delta_tot = V - V_hat = sum_j( delta_j )
```

即 `sum_j delta_j` 不是另起定义，而是由「总体 = 叶子之和」对两列同时成立推出的。**若某一维 rollup 与总体不一致，须先对齐再加总**，否则上式不成立。份额 `c_j = delta_j / delta_tot`（`delta_tot != 0`）用于「覆盖度 / 头部集中度」叙述；**不同维度视角下的 `c_j` 不可简单相加**（见第 5 节）。

**（2）HotSpot：Potential Score（加法 KPI + 多维属性）**

Sun et al., *IEEE Access* 2018, DOI `10.1109/ACCESS.2018.2804764`；定义见原文 IV-B.2 节。LEAF 上实际 `v`、预测 `f`；假设候选集 `S` 为根因时由 **ripple effect** 得反事实 `a(S)`；欧氏距离 `d`。式 (6)(7) 的标量写法：

```
ps(S) = max( 1 - d(v, a(S)) / d(v, f),  0 )
```

`ps` 落在 `[0, 1]`；越大表示「在假设 S 为根因下，叶子层实际值与反推值越接近」。本 skill 不要求复现 MCTS / 剪枝实现；若引用 HotSpot，应说明数据是否满足其加法 KPI 与层次结构假设。

## 4. 默认流程

### Step 1. 先做单维扫描

先对每个维度单独扫描，回答两个问题：
- 哪个维度最可疑
- 哪些维度值在该维度内最异常

这一步的目标是快速收缩搜索空间，而不是直接下最终结论。

可用的打分思路：
- 异常贡献度
- 分布偏移程度
- 加权风险或 weighted risk
- 候选对子异常的覆盖度

须同时考虑**业务权重**与**样本稳定性**；若借鉴 HotSpot / RiskLoc / iDice 等具体打分，仍须落到「幅度 + 覆盖 + 权重 + 稳定」四要素上。

### Step 1.5. 因果链上的候选过滤（条件触发）

**触发**：候选沿前驱→后继（母→子、源→宿等）相互影响，而非独立维度取值；单纯按幅度 / Z-score 排序易把后继误判为根因。本步在 Step 1 后、Step 2 前执行；不触发则跳过。任务是否属本类及未证实传导链时的收口，见 [01 的 3.1 节](../core/01_problem_classification.md)。

- **时间**：在 event 子窗口（[02 的 3.1 节](../core/02_data_preparation.md)）记各候选首次偏离时间；越早越优先；传播时序上明显靠后的标为「受影响实体」。同窗不可分时保留并列候选。
- **证据类型**：产出/行为指标权高；状态/资源次之；**仅**下游后继偏离不得单独定根因。状态与产出方向矛盾 → 降权并查前驱。
- **自洽**：每条证据须有一句方向/机制自洽说明；不自洽或无说明 → 降权或标「证据未解释」，不得作主结论。
- **输出字段**：候选实体、证据等级、关键指标偏离、受影响实体、时间证据、未解释反向信号；结构化输出边界见 [52_output_guardrails.md](../core/52_output_guardrails.md)。

**禁令**：勿仅凭与目标的相关系数或最大异常幅度定唯一根因；因果链未确认时输出候选集并声明依赖未确认；event 未对齐勿用全样本相关排序。

**适用边界**：若候选对象不是维度切片，而是**多个指标节点**（同一对象上的不同指标时序）且节点间有传播 / 依赖结构，转 [17_multivariate_metric_propagation.md](17_multivariate_metric_propagation.md)；本步骤的时间 / 直接信号 / 自洽要素在 17 中被复用，无须重复执行。

### Step 2. 再做低阶组合

只对单维头部维度继续做低阶组合：
- 渠道 × 地区
- 设备 × 版本
- 品类 × 价格带

**默认顺序**：
1. 单维
2. 二阶组合
3. 必要时才考虑更高阶

### Step 3. 必要时做高阶 / 启发式搜索

当下列条件成立时，才升级：
- 单维和二阶组合都无法充分解释
- 怀疑 root cause 是多项组合，不是单点
- 维度空间大到无法手工排查

此时可借鉴 iDice / HotSpot / Squeeze / RiskLoc 的共性做法：
- 用启发式搜索替代全量枚举
- 用剪枝控制搜索空间
- 同时看异常幅度、覆盖度和业务权重
- 允许 root cause 以集合形式输出

### Step 4. 输出候选集合，而不是唯一真因

输出至少包含：
- 单维头部候选
- 低阶组合候选
- 若做了高阶搜索，则输出 top candidate sets
- 对每个候选给出覆盖度、方向、样本量和重叠说明

## 5. Guardrails

本节只补充维度定位方法的专项风险与保守表达，不替代 [52_output_guardrails.md](../core/52_output_guardrails.md) 的结果收敛自检，也不定义额外输出格式。

- 不同维度视角的结果不可直接加总
- 高重叠头部项可能是同一现象的不同投影
- 多维搜索结果默认是候选解释，不自动升级为根因定论
- 小样本组合若异常再大，也不能直接排到头部
- 维度组合越高阶，越要谨慎，越要控制支持度阈值

## 6. 何时优先用 / 何时不要用

| 场景 | 建议 |
|------|------|
| 目标是找哪个维度值或维度组合最可疑 | 优先叠加本文件 |
| 已有主路径结果，但需要进一步定位 blame scope | 叠加本文件 |
| 只有总量，没有维度明细 | 不使用 |
| 只有单一明显头部项 | 单维扫描即可，不必高阶搜索 |
| 目标本来是因果识别 | 升级到 C 分支 |
