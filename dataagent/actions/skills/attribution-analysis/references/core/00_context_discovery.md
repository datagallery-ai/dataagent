# 上下文发现与归因问题对齐

本文件在正式分析前使用，目标是把模糊的业务问题收敛为可检验的分析任务。这是整套 skill 的第一步，跳过此步会导致后续所有方法选型可能答非所问。

## 1. 可行动性审查（最先做）

在展开任何技术分析之前，先回答一个问题：

> 这次归因的结果，会触发什么可逆决策？

| 回答 | 行动 |
|------|------|
| 会触发预算调整 / 投放策略 / 产品改动 / 模型修复 | 启动完整流程，同步规划校准/验证方案 |
| 只是"了解情况" / 信息汇报 | 降低复杂度，用一层拆解 + 简单报告 |
| 不确定 | 先按简单模式做，看结果后再决定是否深入 |

**为什么要先做这步**：没有可执行动作的归因只是解释故事。投入大量建模资源却不改变任何决策，是最常见的资源浪费。

## 2. 对齐句式（必须填写）

每次归因分析开始前，必须写出：

> 在（业务/决策场景）里，我们要把（谁的什么结果）归因给（哪些候选因素/触点/特征），在（时间窗/单位粒度）上，采用（因果/非因果）口径，以支持（什么可执行动作）。

填不完整的部分 = 需要向用户确认的部分。这一句的填空直接决定后续 pipeline 的数据设计与评估方式。

## 3. 五件事确认清单

| # | 要确定的事 | 确认方式 | 填不完整的后果 |
|---|---------|--------|------------|
| 1 | **归因对象**：谁的什么结果 | MCP 元数据 / 用户提供 | 分析方向不明 |
| 2 | **候选因素**：归因给什么 | MCP 元数据 / 用户提供 | 遗漏关键维度或引入无关维度 |
| 3 | **归因口径**：变化解释 / 功劳分配 / 增量效应 | 必须显式判断 | 用错方法体系 |
| 4 | **粒度与时间窗** | 数据可用性决定 | 粒度不匹配导致估计失真 |
| 5 | **可执行动作** | 决定结论需要多"硬" | 过度量化或不足量化 |

## 4. 口径区分（关键决策点）

以下三种口径是不同的分析问题，不可混用：

| 口径 | 含义 | 对应分支 | 典型问法 |
|------|------|---------|---------|
| 变化解释 | 指标为什么变了，变化来自哪里 | A | "GMV 为什么下降了" |
| 功劳分配 | 路径或触点上对结果的功劳拆分（常以营销渠道表述） | B | "各触点/各渠道对转化功劳多少" |
| 增量效应 | 干预导致了多少因果增量 | C | "新策略带来多少增量" |

**常见混淆**：
- 用户说"归因"时，80% 的情况实际是 A（变化解释），不是 C（增量效应）
- 用户说"各渠道贡献"时，可能是 A（维度拆解）也可能是 B（**路径或功劳模型语境**），必须确认是否有**路径或功劳建模**诉求

## 5. 上下文获取优先级协议

### 优先级 1：MCP 数据工具（如果可用）
- 列出可用数据库/表
- 获取表 schema 和列注释
- 采样数据确认格式和分布
- 查询指标平台/语义层获取指标定义

### 优先级 2：Workspace 中的 instructions 文件
- 检查是否有 `.instructions.md` 提供业务语义

### 优先级 3：向用户确认
- 对每一项缺失信息，提出具体问题
- 不假设、不猜测

## 6. 语义歧义必须确认清单

以下歧义不解决就开始分析，结果大概率不可用：

- 指标口径（含税/不含税？UV/PV？自然日/业务日？）
- 去重逻辑（user_id/device_id/account_id？）
- 数据排除规则（测试数据？内部员工？）
- 时区定义
- "转化"的定义（注册？付费？加购？）
- "归因窗"的长度与起止点
- "渠道"的层级定义（一级渠道/子渠道/Campaign？）

## 7. 最小可分析条件

- **必须有**：归因对象 + 口径判断 + 至少一种可拆解结构（公式/维度/路径/特征）
- **缺少任一项时**：说明能做到什么程度，不能做什么
- **口径不明时的保守策略**：先按 A（指标异动/变化解释）处理，不要在口径不明时就选方法

## 8. 本文件与其他 reference 的关系

- 本文件完成后 → 进入 [01_problem_classification.md](01_problem_classification.md) 做分类
- 口径确认结果直接决定后续走哪条分支
