# B0. 通用功劳分配（General Contribution Allocation）

**定位**：本文件服务于 B 分支下的非营销、非路径专用场景。目标是在一个**已发生结果**下，把功劳 / credit / contribution 分配给多个参与因素，例如资源、产品功能、团队、内容位、流程节点、模块或项目参与方。

B0 回答“谁分多少功劳”，不回答“指标为什么变化”（A），也不回答“如果没有某因素会少多少增量”（C）。若需要因果增量口径，应转 [23](23_incrementality_testing.md) 或 C 分支。

## 1. 适用前提

- 已有一个固定结果总量，例如总 GMV、总转化、总产出、成本节省、活跃留存、项目收益或工单解决量。
- 已定义候选贡献因素集合，例如功能、资源、团队、模块、内容位、流程节点、供应方或协作角色。
- 目标是分配 credit / contribution，而不是解释两期变化、群体差异或模型预测。
- 能说明每个候选因素与结果之间的参与证据，例如事件记录、使用记录、投入记录、依赖关系、归属规则或业务账本。

## 2. 入场前确认清单

| 检查项 | 最低要求 | 不满足时处理 |
|------|---------|------------|
| 结果口径 | 明确要分配的总结果、时间窗和单位 | 不清晰则停在 Phase 1 对齐 |
| 候选因素 | 明确哪些因素有资格分功劳 | 先定义 inclusion / exclusion 规则 |
| 分配单位 | 明确按订单、用户、项目、事件、资源、功能或时间段分 | 粒度不清时不得汇总成总份额 |
| 是否可加 | 说明功劳是否要求加总到 100% 或总价值 | 不可加时输出相对排序而非份额 |
| 重叠与协同 | 说明因素是否互斥、可重叠或存在交互 | 有交互时不得只用单因素线性分摊 |
| 决策用途 | 说明是否用于预算、资源、人力或 ROI 决策 | 用于高风险决策时默认叠加 23/C 校准 |

## 3. 方法选择

| 方法 | 适用数据 | 核心假设 | 何时优先用 | 何时不要用 | 输出 |
|------|---------|---------|-----------|-----------|------|
| 规则分配 | 已有业务规则、权重或归属账本 | 规则代表本次 credit 口径 | 会计分摊、团队对齐、快速 baseline | 规则会被误读成数据发现 | allocated value、share、规则说明 |
| 活动量 / 投入量分摊 | 使用次数、时长、成本、人天、曝光、处理量等 | 活动量与功劳近似成比例 | 需要透明、可复核的基线 | 产出质量差异大或存在强交互 | activity-based credit |
| 结果映射分摊 | 每个结果可映射到一个或多个因素 | 映射关系可信，去重规则明确 | 订单-功能、工单-团队、内容-转化等映射清楚 | 归属链大量缺失或重复归因不可控 | mapped contribution |
| Shapley / coalition 分配 | 可定义因素组合的 value function | 边际贡献可按加入顺序公平分配 | 多因素协同明显，需要公平分摊 | value function 无法定义或组合极稀疏 | Shapley credit |
| 回归 / 模型 ablation | 有样本级结果、因素特征和控制变量 | 模型能近似结果生成关系 | 需要数据驱动排序或控制部分混杂 | 样本不足、共线性严重、模型诊断失败 | model-dependent contribution |
| 约束优化分配 | 有总量约束、上下限、业务规则 | 约束能表达真实分配边界 | 需要满足预算、人力、容量等硬约束 | 约束只是事后调数字 | constrained allocation |

## 4. 默认执行流程

### Step 1. 固定功劳口径

- 写清待分配结果：总量、时间窗、单位和是否需要加总到 100%。
- 写清候选因素：纳入条件、排除条件、层级和去重规则。
- 写清分配单位：先在最细粒度分配，再按调用方要求的粒度汇总。

### Step 2. 先做透明 baseline

- 优先使用规则分配、活动量分摊或结果映射分摊。
- baseline 的作用是暴露口径问题，不是证明真实贡献。
- 输出时必须同时给出 raw credit 与 normalized credit；若不能归一化，说明原因。

### Step 3. 处理重叠、协同与模型化贡献

- 因素互斥且结果可直接映射时，可保持简单分配。
- 因素重叠但交互较弱时，可用去重规则、权重规则或回归 / ablation。
- 因素存在明显协同时，优先考虑 Shapley、coalition value 或显式交互项。
- 模型化方法必须检查诊断、共线性、残差和敏感性；契约需要时再输出。

### Step 4. 校验与收口

- 加和校验：若声称分摊总结果，功劳之和必须等于总结果或 100%。
- 稳定性校验：改变权重、窗口、候选集合或聚合层级后，头部因素是否稳定。
- 业务约束校验：贡献方向和量级是否违反已知机制、容量或流程约束。
- 决策校准：若用于预算、资源或 ROI 决策，进入 [23](23_incrementality_testing.md) 或 C 分支校准。

## 5. 输出契约

推荐输出至少包含：

```text
method, entity_type, entity_id, raw_credit, normalized_credit, value_scope, allocation_rule, assumptions, warnings
```

输出语义至少覆盖：

- 待分配结果与时间窗。
- 候选因素集合及排除项。
- 分配方法与选择理由。
- 是否满足加和 / 归一化。
- 不确定性、敏感性或规则依赖。
- 结论是否经过 23/C 的增量校准。

## 6. Guardrails

1. B0 的“贡献”是功劳分配口径，不自动代表因果增量。
2. 不得把两期变化的维度拆解包装成 B0；变化解释应走 A。
3. 规则分配是定义，不是估计；必须写清规则来源。
4. Shapley 的公平分配依赖 value function；未定义 value function 时不得声称做了 Shapley。
5. 模型 ablation 是模型依赖贡献，不等于真实世界因果贡献。
6. 任何用于高风险预算、资源或人力调整的 B0 结论，都应说明如何用 [23](23_incrementality_testing.md) 或 C 分支校准。
